# Landing Page Pattern Reference

Use this reference when a landing page needs stronger design direction, clearer structure, or reference-backed conversion choices. It distills recurring patterns observed across public LPs for developer tools, creative tools, finance, work platforms, AI builders, website builders, and infrastructure platforms.

## Structure Check Template

When analyzing a reference LP, capture:

1. Hero promise: what outcome, category, or campaign idea leads the page.
2. CTA split: primary action, secondary action, and whether the CTA is a workflow input, product action, download, signup, or sales motion.
3. First-screen proof: product UI, output gallery, workflow state, logo strip, metric, customer quote, security claim, or compliance note.
4. Section order: the actual narrative path from hero to final CTA.
5. Proof and trust: logos, named customers, quotes, metrics, ratings, uptime, compliance, funding, press, templates, marketplace, or ecosystem.
6. Product visualization: screenshot, demo, workflow simulation, bento map, customer outcome gallery, platform diagram, or core input surface.
7. Interaction and motion: tabs, carousels, scroll-led product panels, prompt boxes, video, calculators, demos, or animated product states.
8. SEO posture: title, H1, meta description, index/noindex clues, canonical, structured data, comparison pages, solution pages, templates, docs, and footer taxonomy.

## Proof Modes

- Workflow simulation: best for operational, developer, productivity, and AI tools. Show a recognizable task in progress with real-looking objects, statuses, assignees, prompts, outputs, timestamps, or integrations.
- Live product or demo surface: best when the product's interaction is the selling point. Put the editor, terminal, prompt box, dashboard, or app composer in the hero or first section.
- Customer outcome gallery: best for creative tools, website builders, marketplaces, and services where the result is more persuasive than the tool chrome.
- Quantified enterprise proof: best for B2B platforms and high-consideration tools. Use metrics, named customers, uptime, adoption counts, savings, or performance outcomes early.
- Ecosystem or taxonomy proof: best for platforms. Let navigation, product families, integrations, templates, docs, and resource depth demonstrate breadth.
- Trust and compliance proof: best for finance, healthcare, security, and regulated categories. Pair conversion CTAs with disclosures, security links, compliance, insurance, or risk-reduction language.

## Section Archetypes

- Workflow lifecycle: hero -> proof -> intake/start -> plan/build -> collaborate/review -> monitor/measure -> trust -> final CTA. Useful for product operations, developer workflow, and AI collaboration tools.
- Build path segmentation: hero -> trust -> "how do you want to build?" choices -> product paths -> outcomes -> final CTA. Useful for multi-intent builders, website platforms, and creation suites.
- Product surface map: hero -> logo proof -> job-based bento modules -> deeper features -> customer story -> metrics -> final CTA. Useful for broad work platforms and multiproduct collaboration tools.
- Customer outcome gallery: hero -> example gallery -> creation workflow -> feature proof -> customer stories -> resources -> final CTA. Useful for visual creation tools, agencies, marketplaces, and website builders.
- Infrastructure credibility: hero -> core jobs -> global scale metrics -> buyer segments -> developer/platform proof -> news/resources -> final CTA. Useful for financial, API, payments, infrastructure, and platform businesses.
- Trust-first conversion: hero with inline signup -> compliance note -> product proof -> testimonials -> economics -> security -> final CTA. Useful for regulated, finance, healthcare, security, and compliance-heavy products.
- Minimal developer platform: hero -> exact deploy/build CTA -> three-step flow -> platform depth -> enterprise/security -> docs/resources. Useful for developer infrastructure and deployment platforms.

## Patterns From Reference LP Types

### Product Operations Workflow

- Use a realistic product workflow as the hero visual, with named objects, statuses, labels, integrations, and handoffs.
- Organize sections by lifecycle rather than a flat feature list.
- Support future-facing positioning with present-tense proof: changelog freshness, named customers, usage counts, and security links.

### AI Developer Tool

- Show the product completing a recognizable development task above the fold.
- Use CTA specificity, such as a platform-specific download, when the action is concrete.
- Pair practitioner credibility with technical freshness: research, changelog, docs, or ecosystem.

### Visual Creation Platform

- For visual creation tools, lead with published customer outcomes before editor UI.
- Make the secondary CTA another creation mode, such as "Start with AI," when that lowers friction.
- Treat templates, comparisons, marketplace, and feature pages as part of the SEO architecture.

### Financial Infrastructure Platform

- Lead with a broad category promise, then immediately scope it with concrete jobs-to-be-done.
- Place credibility math early: processed volume, uptime, currencies, named customers, or adoption.
- Branch later sections by buyer type, such as startups, enterprises, platforms, and developers.

### Regulated Business Account Product

- Pair a bold emotional headline with a concrete operational claim.
- Use an embedded workflow CTA, such as email capture or account start, when signup is the conversion.
- Alternate feature proof with economic proof, such as savings, yield, cashback, fee reduction, or time saved.

### Broad AI Workspace

- A campaignable hero can work if it is immediately translated into plain product outcomes.
- Put logo trust before deep feature explanation in crowded categories.
- Use job-based bento modules, each with one short promise and one real product surface.

### AI App Builder

- Put the product's core input directly in the hero when the input is the product experience.
- Explain workflow in three verbs: start, watch, refine/ship.
- Pair social proof with templates or example outputs so visitors see both adoption and attainable results.

### Website Growth Platform

- Lead with outcome language, then segment intent with build-path choices.
- Convert proof into metric cards, not only testimonials.
- Mirror the visible promise in metadata, structured data, solution pages, and footer taxonomy.

### Developer Deployment Platform

- For developer products, compress the promise into a workflow such as build -> deploy -> scale.
- Let product taxonomy and ecosystem depth act as proof when screenshots would under-explain the platform.
- Use primary CTA verbs that match the actual job, such as "Deploy" rather than generic "Learn more."

## Implementation Checklist

- The first viewport should answer: what is this, who is it for, what can I do next, and what proof makes the claim believable.
- The primary CTA should name the actual next job. Avoid generic labels when a precise action is available.
- The secondary CTA should serve a distinct intent: demo, sales, docs, template, download, or AI/start mode.
- Each major claim should have nearby proof. Do not save all credibility for the footer.
- Visuals should show product state, output, workflow, metrics, or trust evidence. Avoid purely atmospheric art unless brand storytelling is the actual product.
- Section order should follow the buyer's decision path, not the internal feature taxonomy.
- For broad platforms, segment by buyer type, job, or build path before visitors feel lost.
- For organic pages, keep title, H1, meta description, and visible copy aligned around category, audience, and wedge.
- For campaign or paid pages, decide whether to noindex, canonicalize, or keep a lean indexable surface.
